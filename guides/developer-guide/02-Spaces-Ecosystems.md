# Spaces & Ecosystems (Developer Architecture)

This document outlines the technical implementation of "Spaces" (Zones and Locations) within FloraSync. Spaces form the backbone of the physical hierarchy, allowing for localized microclimates and intelligent batch processing without overloading database relations or journal logs.

## 1. Data Model & Hierarchy

The physical space structure is strictly two-tiered to maintain predictability and simplify UI traversal.

```typescript
// types.ts
export interface Zone {
  id: string;
  name: string;
  description?: string;
  imageUrl?: string;
  evaporationModifier?: number; // Microclimate modifier (e.g., 0.8 for humid, 1.2 for dry)
  isCovered?: boolean;          // Rain exemption flag
  journal?: JournalEntry[];
  isPinned?: boolean;
  pinnedActions?: Record<string, string[]>;
}

export interface Location {
  id: string;
  name: string;
  zoneId: string;              // Belongs to one Zone
  isPinned?: boolean;
  journal?: JournalEntry[];
  imageUrl?: string;
  pinnedActions?: Record<string, string[]>;
}

export interface PlantInstance {
  // ...
  locationId: string;          // Plants belong to Locations, NOT directly to Zones
  // ...
}
```

*   **Zones (Macro-Environments):** Represent overarching areas like a "Greenhouse". They contain `evaporationModifier` to globally affect watering schedules of all descendant plants, and `isCovered` to opt out of global weather events (like rain logging).
*   **Locations (Micro-Environments):** Represent specific shelves or beds. Plants (`PlantInstance`) are strictly tied to a `locationId`. They inherit the microclimate and weather restrictions of their parent Zone.

## 2. Trickle-Down Architecture for Batch Actions

When a user scans a Zone or Location QR code or uses a UI button to perform a batch action (e.g., "Water All"), FloraSync utilizes a "Trickle-Down" state update strategy.

**Implementation detail (`src/hooks/useGardenState.ts`):**
Instead of writing an individual `JournalEntry` for every single plant affected, the engine:
1.  **State Update:** Iterates through all child `PlantInstance`s and updates their relevant metric (e.g., `lastWatered: new Date().toISOString()`).
2.  **Journal Rollup:** Appends a *single* `JournalEntry` to the parent entity (`Zone.journal` or `Location.journal`).

This prevents "spamming" plant histories, scales well, and significantly reduces the payload size and render overhead for the Global Journal and Plant Detail views.

## 3. QR Code & Routing System

The "Zero-Click" QR system is built entirely on standard URL routing. QR codes generated by the app encode internal relative or absolute URLs (e.g., `/location/:id` or `/zone/:id/water-all`).

**Implementation detail (`src/AppRouter.tsx`):**
*   The `Scanner` component passes the raw decoded string to `handleScanResult`.
*   The string is parsed as a URL against `window.location.origin`.
*   If the path matches a known entity type (`['plant', 'location', 'zone']`), the app pushes the state using the History API (`window.history.pushState`) and the main router synchronizes the active view.
*   Optional action parameters (e.g., `/zone/z-123/water`) are caught by the router and passed down as `initialAction` props to `LocationDetail` or `ZoneDetail`.
*   Components like `LocationDetail` consume `initialAction` in a `useEffect` on mount to immediately execute the appropriate batch hook (e.g., `onBatchWaterLocation`).

## 4. Ecosystem & Microclimate Modifiers

The `evaporationModifier` is applied within the hydration math engine. If a Zone has an `evaporationModifier` of `1.2`, a plant that normally needs watering every 10 days will have its dynamic threshold adjusted.

**Extending Ecosystem Logic:**
*   **State:** Extend the `Zone` interface in `types.ts`.
*   **Logic:** Update the hydration and feed calculation utilities to ingest the new modifier.
*   **Cascade:** Ensure functions like those in `useGardenState` properly cascade the modifier to child locations and plants during evaluation.
